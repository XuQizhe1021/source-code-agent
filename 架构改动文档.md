# 架构改动文档

## 步骤1：Endpoint Thin Controller（本次迭代）

### 本次目标
将 `graph` 业务域中的“创建知识图谱”主流程从 Endpoint 下沉到 application service，让控制器只保留协议层职责，减少耦合与复杂度。

### 代码改动清单（文件路径 + 职责变化）
- `app/application/graph/__init__.py`：新增应用服务包入口，承载图谱用例编排模块。
- `app/application/graph/graph_app_service.py`：新增 `GraphAppService` 与 `CreateGraphCommand`，集中处理“创建图谱 + 尝试创建 Neo4j 子图”业务编排。
- `app/api/v1/endpoints/graph.py`：`POST /` 改为薄控制器，仅做参数绑定、调用应用服务、异常映射。

### 核心实现说明（含关键注释点）
- 在 `GraphAppService.create_graph()` 中统一完成图谱创建事务，避免 Endpoint 直接操作数据库细节。
- 在 `GraphAppService._try_create_neo4j_subgraph()` 中保留“Neo4j 子图失败不阻塞主流程”的兼容策略，确保前端调用体验不被增强能力拖垮。
- 在 Endpoint 内通过 `get_graph_app_service()` 依赖注入服务实例，控制器只做输入输出适配。
- 关键代码均补充了通俗中文注释，重点说明了“为什么拆到服务层”和“为什么异常只记录不抛出”。

### 达到的效果（可维护性/解耦/性能）
- 可维护性：`POST /graphs/` 的核心业务编排不再散落在路由函数内，改动面收敛。
- 解耦：控制器与业务流程解耦，为后续引入端口适配器、埋点、重试策略提供稳定落点。
- 性能：本次不改变核心算法路径，运行时性能基本持平；但后续可在服务层统一加缓存/限流而不改 Endpoint。

### 验证记录
- 静态诊断：`GetDiagnostics` 返回空列表（未发现新增 IDE 诊断错误）。
- 最小编译校验：执行  
  `python -m py_compile app/application/graph/graph_app_service.py app/api/v1/endpoints/graph.py`  
  退出码 `0`，通过。

### 遗留问题与下步计划
- 遗留问题：`graph.py` 仍有大量路由包含业务编排（如文件处理、抽取任务等），尚未下沉。
- 下步计划：在步骤2中按同样模式继续拆分一个高耦合流程（建议 `/{graph_id}/extract` 或 `/{graph_id}/files/upload`），逐步把 Endpoint 压薄。

## 步骤1：Endpoint Thin Controller（本次迭代-2）

### 本次目标
继续在 `graph` 业务域做同域增量拆分，将 `POST /{graph_id}/extract` 的业务编排下沉到应用服务，进一步压薄 Endpoint。

### 代码改动清单（文件路径 + 职责变化）
- `app/application/graph/graph_app_service.py`：新增 `ExtractKnowledgeCommand` 与 `submit_extract_task()`，承接抽取流程中的校验、状态推进、任务创建与后台任务投递。
- `app/api/v1/endpoints/graph.py`：`/{graph_id}/extract` 改为薄控制器，仅做参数绑定与调用应用服务。

### 核心实现说明（含关键注释点）
- `submit_extract_task()` 统一处理图谱存在性、模型绑定、`fileId` 参数、文件归属等校验，避免 Endpoint 重复堆砌判断。
- 保持兼容策略：图谱未绑定模型时仍返回原业务结构（`code/message`），不改变前端处理方式。
- 兼容旧行为：文件状态为 `pending/failed` 时先投递 `process_graph_file`，再创建抽取任务并投递 `extract_knowledge_with_llm_task`。
- 关键处补充了通俗中文注释，明确“为什么该逻辑属于应用服务编排”。

### 达到的效果（可维护性/解耦/性能）
- 可维护性：`extract` 路由从多段数据库/任务编排逻辑收敛为服务调用，复杂度显著下降。
- 解耦：Controller 与任务编排细节解耦，后续调整抽取流程时无需改 Endpoint 协议层代码。
- 性能：本次未增加额外 I/O 路径，性能表现与原流程基本一致。

### 验证记录
- 静态诊断：`GetDiagnostics` 返回空列表。
- 最小编译校验：执行  
  `python -m py_compile app/application/graph/graph_app_service.py app/api/v1/endpoints/graph.py`  
  退出码 `0`，通过。

### 遗留问题与下步计划
- 遗留问题：`/{graph_id}/files/upload` 仍含上传循环与失败聚合逻辑，尚未下沉。
- 下步计划：继续步骤1同域拆分 `/{graph_id}/files/upload`，把批量上传编排迁移到 `GraphAppService`，Endpoint 仅保留输入输出适配。

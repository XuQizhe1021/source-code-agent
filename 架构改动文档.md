# 架构改动文档

## 步骤1：Endpoint Thin Controller（本次迭代）

### 本次目标
将 `graph` 业务域中的“创建知识图谱”主流程从 Endpoint 下沉到 application service，让控制器只保留协议层职责，减少耦合与复杂度。

### 代码改动清单（文件路径 + 职责变化）
- `app/application/graph/__init__.py`：新增应用服务包入口，承载图谱用例编排模块。
- `app/application/graph/graph_app_service.py`：新增 `GraphAppService` 与 `CreateGraphCommand`，集中处理“创建图谱 + 尝试创建 Neo4j 子图”业务编排。
- `app/api/v1/endpoints/graph.py`：`POST /` 改为薄控制器，仅做参数绑定、调用应用服务、异常映射。

### 核心实现说明（含关键注释点）
- 在 `GraphAppService.create_graph()` 中统一完成图谱创建事务，避免 Endpoint 直接操作数据库细节。
- 在 `GraphAppService._try_create_neo4j_subgraph()` 中保留“Neo4j 子图失败不阻塞主流程”的兼容策略，确保前端调用体验不被增强能力拖垮。
- 在 Endpoint 内通过 `get_graph_app_service()` 依赖注入服务实例，控制器只做输入输出适配。
- 关键代码均补充了通俗中文注释，重点说明了“为什么拆到服务层”和“为什么异常只记录不抛出”。

### 达到的效果（可维护性/解耦/性能）
- 可维护性：`POST /graphs/` 的核心业务编排不再散落在路由函数内，改动面收敛。
- 解耦：控制器与业务流程解耦，为后续引入端口适配器、埋点、重试策略提供稳定落点。
- 性能：本次不改变核心算法路径，运行时性能基本持平；但后续可在服务层统一加缓存/限流而不改 Endpoint。

### 验证记录
- 静态诊断：`GetDiagnostics` 返回空列表（未发现新增 IDE 诊断错误）。
- 最小编译校验：执行  
  `python -m py_compile app/application/graph/graph_app_service.py app/api/v1/endpoints/graph.py`  
  退出码 `0`，通过。

### 遗留问题与下步计划
- 遗留问题：`graph.py` 仍有大量路由包含业务编排（如文件处理、抽取任务等），尚未下沉。
- 下步计划：在步骤2中按同样模式继续拆分一个高耦合流程（建议 `/{graph_id}/extract` 或 `/{graph_id}/files/upload`），逐步把 Endpoint 压薄。

## 步骤1：Endpoint Thin Controller（本次迭代-2）

### 本次目标
继续在 `graph` 业务域做同域增量拆分，将 `POST /{graph_id}/extract` 的业务编排下沉到应用服务，进一步压薄 Endpoint。

### 代码改动清单（文件路径 + 职责变化）
- `app/application/graph/graph_app_service.py`：新增 `ExtractKnowledgeCommand` 与 `submit_extract_task()`，承接抽取流程中的校验、状态推进、任务创建与后台任务投递。
- `app/api/v1/endpoints/graph.py`：`/{graph_id}/extract` 改为薄控制器，仅做参数绑定与调用应用服务。

### 核心实现说明（含关键注释点）
- `submit_extract_task()` 统一处理图谱存在性、模型绑定、`fileId` 参数、文件归属等校验，避免 Endpoint 重复堆砌判断。
- 保持兼容策略：图谱未绑定模型时仍返回原业务结构（`code/message`），不改变前端处理方式。
- 兼容旧行为：文件状态为 `pending/failed` 时先投递 `process_graph_file`，再创建抽取任务并投递 `extract_knowledge_with_llm_task`。
- 关键处补充了通俗中文注释，明确“为什么该逻辑属于应用服务编排”。

### 达到的效果（可维护性/解耦/性能）
- 可维护性：`extract` 路由从多段数据库/任务编排逻辑收敛为服务调用，复杂度显著下降。
- 解耦：Controller 与任务编排细节解耦，后续调整抽取流程时无需改 Endpoint 协议层代码。
- 性能：本次未增加额外 I/O 路径，性能表现与原流程基本一致。

### 验证记录
- 静态诊断：`GetDiagnostics` 返回空列表。
- 最小编译校验：执行  
  `python -m py_compile app/application/graph/graph_app_service.py app/api/v1/endpoints/graph.py`  
  退出码 `0`，通过。

### 遗留问题与下步计划
- 遗留问题：`/{graph_id}/files/upload` 仍含上传循环与失败聚合逻辑，尚未下沉。
- 下步计划：继续步骤1同域拆分 `/{graph_id}/files/upload`，把批量上传编排迁移到 `GraphAppService`，Endpoint 仅保留输入输出适配。

## 步骤2：治理万能 utils，落实 DIP（本次迭代）

### 本次目标
选择 `knowledge` 主链路，以 `POST /api/knowledge/{knowledge_id}/retrieve` 为落地点，完成“依赖抽象 + 适配器注入”的最小闭环，减少应用层对 `app/utils/knowledge.py` 中具体实现的直接依赖。

### 代码改动清单（文件路径 + 职责变化）
- `app/application/knowledge/ports.py`：新增端口抽象，定义 `KnowledgeRepositoryPort`（数据访问）与 `EmbeddingGatewayPort`（外部网关）。
- `app/application/knowledge/knowledge_retrieval_service.py`：新增检索用例服务 `KnowledgeRetrievalService`，仅依赖端口，不直接 import ORM/SDK。
- `app/adapters/knowledge/repositories.py`：新增 `SqlAlchemyKnowledgeRepository`，提供 Repository 端口实现。
- `app/adapters/knowledge/gateways.py`：新增 `ModelEmbeddingGateway`，封装模型向量化与向量检索网关实现。
- `app/api/v1/endpoints/knowledge.py`：`/{knowledge_id}/retrieve` 改为调用应用服务，通过依赖注入组装 adapter。
- `tests/services/test_knowledge_retrieval_service.py`：新增 fake adapter 测试，验证服务层可脱离真实基础设施测试。

### 核心实现说明（含关键注释点）
- 在端口定义处添加中文注释说明“为什么抽象”：高层不再绑定 SQLAlchemy/向量库 SDK，便于替换实现与单测隔离。
- `KnowledgeRetrievalService` 负责参数归一化、业务校验、结果过滤与拼装，体现“应用层编排”职责。
- Endpoint 仅做依赖注入与异常映射，维持原 API 路径与返回结构兼容。

### 达到的效果（可维护性/解耦/性能）
- 可维护性：检索业务规则集中到应用服务，改动范围收敛。
- 解耦：应用层对具体 ORM/向量库调用的直接依赖被端口隔离，满足 DIP。
- 性能：本次是结构治理，主链路执行路径基本不变，运行性能预期持平。

## 步骤3：建立业务域通信边界（本次迭代）

### 本次目标
围绕 Agent 与 Knowledge 的交互链路改造，先落地“Agent 检索知识库增强”主路径：引入 ACL 防腐层、DTO 契约和最小事件总线，避免 Agent 域直接读取 Knowledge 域内部细节。

### 代码改动清单（文件路径 + 职责变化）
- `app/application/acl/contracts.py`：新增跨域 DTO `KnowledgeSnippetDTO`，收敛 Agent<->Knowledge 交换字段。
- `app/application/acl/knowledge_acl.py`：新增 `KnowledgeACL`，作为 Agent 访问 Knowledge 的唯一跨域入口。
- `app/application/events/event_bus.py`：新增最小进程内事件总线 `InProcessEventBus`。
- `app/application/events/subscribers.py`：新增默认订阅者注册，处理 `knowledge.retrieved` 事件（当前先做可观测日志动作）。
- `app/services/retrieval_strategies.py`：`KnowledgeRetrievalStrategy` 改为调用 `KnowledgeACL`，移除对 Knowledge 域内部查询/向量细节的直接依赖。
- `app/services/chat_orchestrator.py`：初始化时注册默认事件订阅者。
- `tests/services/test_retrieval_strategies.py`：更新策略测试为 ACL 契约模式。
- `tests/services/test_knowledge_acl.py`：新增 ACL DTO 映射与事件发布测试。

### 核心实现说明（含关键注释点）
- 通过 `KnowledgeACL.retrieve_for_agent()` 把跨域通信收敛为显式接口，Agent 域不再直接 import `app.utils.knowledge` 的内部能力。
- 通过 `KnowledgeSnippetDTO` 明确契约边界，避免跨域共享内部模型对象导致隐式耦合。
- 在 ACL 内发布 `knowledge.retrieved` 事件，把“检索完成后的后续动作”改为事件触发，降低跨域硬编码。
- 关键位置补充了通俗中文注释，明确“域边界”和“通信契约”的改造原因。

### 达到的效果（可维护性/解耦/性能）
- 可维护性：跨域调用路径从“策略直连知识内部实现”变为“策略 -> ACL -> 应用服务/适配器”，可读性提升。
- 解耦：Agent 域与 Knowledge 域内部结构解耦，后续 Knowledge 内部重构对 Agent 影响显著降低。
- 性能：本次为结构治理，核心检索算法和外部调用次数保持不变，性能预期持平。

## 步骤4：优化非关键路径性能，降低主链路阻塞（本次迭代）

### 本次目标
在 `knowledge` 检索主链路落地最小性能闭环：将统计上报改为异步、为高频读请求增加短 TTL 缓存、补充慢依赖超时与降级，减少主请求阻塞时间。

### 代码改动清单（文件路径 + 职责变化）
- `app/application/knowledge/knowledge_retrieval_service.py`：新增进程内 TTL 缓存、嵌入与向量检索超时降级、文件名查询去重缓存。
- `app/api/v1/endpoints/knowledge.py`：`/{knowledge_id}/retrieve` 接口新增 `BackgroundTasks`，将检索统计上报放到响应后异步执行。
- `tests/services/test_knowledge_retrieval_service.py`：新增缓存命中与超时降级测试，验证性能与稳定性策略。

### 核心实现说明（含关键注释点）
- 高频读缓存：对 `knowledge_id + query + threshold + top_k + model_id` 构建稳定 key，命中后直接返回缓存结果，减少重复模型推理与向量检索。
- 异步非关键路径：统计上报使用 `background_tasks.add_task`，即使上报失败也只记录 warning，不影响接口成功返回。
- 超时与降级：嵌入推理与向量检索都增加超时控制，超时后返回可解释的空结果，避免慢依赖拖垮主链路。
- 重复查询优化：同一次检索内对 `file_id -> 文件名` 做本地去重缓存，避免命中列表中的重复 `file_id` 触发重复查询。
- 关键处均补充了通俗中文注释，说明“为什么异步化/缓存/超时降级能提升稳定性”。

### 达到的效果（可维护性/解耦/性能）
- 可维护性：性能策略集中在应用服务，不需要在 endpoint 中散落优化细节。
- 解耦：主流程与统计上报解耦，非关键动作故障不会传播到主业务返回。
- 性能：在重复查询场景，命中缓存时可跳过模型推理与向量检索两段慢依赖，主链路步骤明显减少。

- 遗留问题：当前缓存为进程内短 TTL，未覆盖多实例共享缓存场景；统计上报暂为日志级聚合，未落地独立指标存储。

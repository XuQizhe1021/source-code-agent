# 架构优化建议

## 第一部分：问题诊断（列表形式）

- 问题1：API Endpoint 承担过多职责，形成“上帝模块”
  - 典型位置：`app/api/v1/endpoints/graph.py`、`app/api/v1/endpoints/knowledge.py`、`app/api/v1/endpoints/files.py`、`app/api/v1/endpoints/dashboard.py`。
  - 现象：同一文件同时处理鉴权、参数校验、事务控制、领域规则、外部服务调用、状态机更新与响应拼装。
  - 影响：改一个需求会波及多段逻辑，回归成本高；路径过长导致单次请求延迟波动更大，且难以做局部性能优化。

- 问题2：`app/utils` 变成“万能工具层”，业务逻辑与基础设施耦合
  - 典型位置：`app/utils/graph.py`、`app/utils/knowledge.py`、`app/utils/agent.py`。
  - 现象：同一模块内混合 ORM 读写、文件 I/O、向量化、模型调用、Neo4j 访问、业务规则判断。
  - 影响：高层模块直接依赖具体实现，违反依赖倒置；单元测试需要真实数据库/对象存储/模型环境，测试慢且脆弱。

- 问题3：业务域之间存在跨层直连与跨域直取
  - 典型位置：`agents`、`graph`、`knowledge` Endpoint 与 `utils` 直接互调。
  - 现象：调用链“Endpoint -> utils -> models/db/third-party”跨越多层，且存在在一个域中直接拼装另一个域的数据语义。
  - 影响：模块边界模糊，跨域变更需要联动修改；后续拆分服务或引入独立部署边界时迁移成本高。

- 问题4：非关键路径存在同步阻塞，吞吐与响应时间受限
  - 典型场景：上传后处理、向量化、统计计算、历史记录写入、连接建立与断开。
  - 现象：主请求里仍有较多可延后动作；部分流程重复建立连接、重复查询和序列化。
  - 影响：P95/P99 延迟偏高；高并发下线程/协程被占用，影响核心对话链路吞吐。


## 第二部分：重构实施建议（分步陈述）

### 第1步：按业务能力拆分“上帝 Endpoint”

- 动作名称：应用层收敛（Endpoint Thin Controller）
- 伪代码/目录结构示例（改动前 -> 改动后）

```text
# 改动前
app/
  api/v1/endpoints/
    graph.py         # 鉴权+参数+业务+DB+Neo4j+文件处理
    knowledge.py     # 鉴权+参数+业务+DB+向量化+文件处理

# 改动后
app/
  interfaces/http/
    graph_controller.py      # 只做协议转换和鉴权
    knowledge_controller.py
  application/
    graph/
      graph_app_service.py   # 用例编排
    knowledge/
      knowledge_app_service.py
```

```python
# 改动前（示意）
@router.post("/graphs/{id}/files/upload")
async def upload(...):
    check_permission()
    graph = db.query(...)
    save_file()
    call_neo4j()
    update_status()
    return resp

# 改动后（示意）
@router.post("/graphs/{id}/files/upload")
async def upload(cmd: UploadGraphFileCmd, svc: GraphAppService = Depends(...)):
    return await svc.upload_file(cmd)
```

- 性能与解耦收益说明
  - Endpoint 仅承担输入输出适配，变更面收敛，维护成本明显下降。
  - 可在应用服务层统一加缓存、限流、熔断与指标埋点，避免重复逻辑。
  - 复杂路径可单独压测和优化，不再受巨型路由文件牵制。


### 第2步：引入端口-适配器（Ports & Adapters）替代 `utils` 直连

- 动作名称：抽离仓储与外部网关接口（依赖倒置）
- 伪代码/目录结构示例（改动前 -> 改动后）

```text
# 改动前
application/domain -> app/utils/*.py -> SQLAlchemy/Neo4j/MinIO/LLM

# 改动后
application/domain -> ports(interface) -> adapters(impl) -> infra SDK/ORM
```

```python
# 改动后接口（示意）
class GraphRepository(Protocol):
    def get(self, graph_id: str) -> GraphAggregate: ...
    def save(self, graph: GraphAggregate) -> None: ...

class VectorStorePort(Protocol):
    async def upsert(self, docs: list[DocChunk]) -> None: ...

class LlmPort(Protocol):
    async def chat(self, messages: list[dict], config: dict) -> AsyncIterator[str]: ...
```

- 性能与解耦收益说明
  - 高层模块不再依赖具体 SDK/ORM，实现可替换（本地/云、Mock/真实）。
  - 可对 `VectorStorePort`、`LlmPort` 做连接池、批处理、超时重试策略统一治理。
  - 单元测试直接注入 Fake Adapter，测试执行更快、覆盖更稳定。


### 第3步：建立业务域通信边界（防腐层 + 事件）

- 动作名称：抽离防腐层（ACL）并事件化跨域协作
- 伪代码/目录结构示例（改动前 -> 改动后）

```text
# 改动前
Agent域 直接读取 Knowledge/Graph 域内部模型字段并拼装提示词

# 改动后
Agent域 -> KnowledgeACL / GraphACL -> 域内公开DTO
Agent域 <- DomainEventBus <- KnowledgeFileProcessed / GraphExtracted
```

```python
# 改动后（示意）
class KnowledgeACL:
    async def retrieve_context(self, query: str, kb_ids: list[str]) -> list[KnowledgeSnippet]:
        ...

class GraphACL:
    async def retrieve_subgraph(self, query: str, graph_ids: list[str]) -> GraphContext:
        ...

event_bus.publish(KnowledgeFileProcessed(file_id=..., knowledge_id=...))
```

- 性能与解耦收益说明
  - 禁止跨域共享内部结构，领域边界稳定后，迭代不会互相“牵一发动全身”。
  - 事件驱动让耗时流程异步化，主链路只依赖必要数据，响应更快。
  - 后续拆分微服务时可平滑迁移为消息总线，不必重写领域协议。


### 第4步：将核心规则“去框架化”，沉淀为纯领域模块

- 动作名称：领域模型纯化（Framework Agnostic Domain）
- 伪代码/目录结构示例（改动前 -> 改动后）

```text
# 改动后建议目录
app/
  domain/
    graph/
      entities.py
      services.py
      policies.py
    knowledge/
      entities.py
      services.py
      policies.py
```

```python
# 领域服务（示意，不含 FastAPI/ORM/SDK）
class GraphExtractionPolicy:
    def can_extract(self, graph_status: str, file_status: str) -> bool:
        return graph_status == "active" and file_status == "processed"
```

- 性能与解耦收益说明
  - 领域规则不再绑死框架，重用与测试成本更低。
  - 可对热点规则做纯内存优化（例如策略对象缓存），不触碰基础设施层。
  - 框架升级/替换对核心业务影响最小，长期维护更稳。


### 第5步：性能专项改造（异步化 + 读写分离 + 缓存）

- 动作名称：非关键路径异步化与查询层优化
- 伪代码/目录结构示例（改动前 -> 改动后）

```python
# 改动前（示意）
await process_upload()
await embedding()
db.commit()
return ok

# 改动后（示意）
job_id = await command_bus.enqueue(ProcessUploadAndEmbedding(...))
return {"accepted": True, "job_id": job_id}
```

```text
# 新增建议
application/
  jobs/
    knowledge_jobs.py
    graph_jobs.py
infrastructure/
  cache/
    redis_cache.py
  readmodel/
    dashboard_read_repo.py
```

- 性能与解耦收益说明
  - 非关键链路（通知、日志、向量化、统计汇总）异步化后，主请求时延显著下降。
  - Dashboard/列表查询使用读模型和缓存，减少复杂聚合对主库冲击。
  - 通过读写分离降低锁竞争；缓存命中后可显著降低重复查询开销。


## 优先级与落地节奏（建议）

- Phase 1（1~2周）：第1步 + 第2步（先拆边界、再引入接口），保证“可维护性止血”。
- Phase 2（1~2周）：第3步（ACL+事件）与第5步中“异步化”子集，先优化主链路延迟。
- Phase 3（1周）：第4步 + 第5步中“读写分离/缓存”，完成性能与演进能力收口。


## 验收指标（建议量化）

- 代码结构：单个 Endpoint 文件控制在 300 行以内；业务逻辑文件圈复杂度下降 30%+。
- 解耦程度：应用层对 `adapters` 零直接依赖，全部通过 `ports` 注入。
- 性能目标：核心聊天接口 P95 下降 20%+；上传触发接口平均响应下降 40%+。
- 稳定性：外部依赖异常时，主链路可降级返回，错误隔离不扩散。

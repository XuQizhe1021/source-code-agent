# 实验二：ADT 与 OOP 设计与扩展

## 1. 实验目标
本实验旨在让您在真实的业务代码中深度应用面向对象编程（OOP）和抽象数据类型（ADT）。这是重塑项目架构的第二步，您将学习到：
1. **抽象数据类型（ADT）**：设计保护自身内部状态的数据结构，理解表示不变量（RI）和防御性编程（Defensive Programming）。
2. **面向对象的扩展（OOP）**：深入体验契约式设计，学会利用多态（Polymorphism），遵循里氏替换原则（LSP）和开闭原则（OCP）来扩展系统功能。

## 2. 实验背景
CogmAIt 项目中的 `app/providers/` 提供了一个“可插拔式”的模型接入架构的雏形。任何新的 LLM 服务商只需继承 `ModelProvider` 并实现其定义的方法，系统即可调用。请注意，在当前仓库中，Provider 通常是由 `manager.py` 扫描 `app/providers/` 目录后自动识别和加载的，而不是手工维护一张固定的注册表。

但同样地，系统在很多地方（尤其是智能体聊天主链路中的消息上下文构造）充斥着极其裸露的数据传递机制。诸如列表 `[]`、字典 `{}` 这种毫无封装的数据结构在复杂的流程网络中穿梭，一旦哪一个中间环节手滑修改了数据，就会引发灾难。这就需要引入 ADT 的理念进行改造。

## 3. 实验任务

### 任务 1：OOP 扩展与契约遵守
在这一步，你将化身框架的“使用者”。通过尽量少改动原有核心逻辑的方式，让系统支持一个全新的大模型，体会“面向接口编程”的威力。

**详细操作步骤**：
1. **先确认当前真正生效的代码路径**：
   - 建议先查看 `app/main.py` 和 `app/api/v1/api.py`，确认当前真正生效的主路由。
   - 在当前仓库中，和本实验最相关的主路径通常是 `app/api/v1/endpoints/agents.py`、`app/api/v1/endpoints/models.py`、`app/providers/base.py`、`app/providers/manager.py`、`app/utils/model.py`。
2. **研读基类契约**：
   - 打开 `app/providers/base.py` 文件。
   - 找到 `ModelProvider` 这个类（它继承自 `ABC`，意味着它是个抽象基类）。
   - 仔细阅读里面以 `@abstractmethod` 装饰的方法。当前真实接口名通常是 `chat_completion`、`text_completion`、`embedding` 等。**这些就是新模型服务商必须遵守的“契约”**。如果不实现它们，Python 解释器连实例化都不会允许。
3. **创建你的 Provider**：
   - 进入 `app/providers/` 目录。
   - 新建一个 Python 文件，比如命名为 `deepseek_provider.py` 或者 `zhipu_provider.py`。
   - 在新文件中，引入抽象基类：`from app.providers.base import ModelProvider`
   - 编写你的类（比如 `class DeepSeekProvider(ModelProvider):`），并开始实现那些被 `@abstractmethod` 标记的方法。
   - **🤖 探索：使用 AI 辅助**
     你可以复制 `base.py` 里的接口契约，告诉 AI 助手：“`请帮我根据这个 ModelProvider 接口，用 Python 的 httpx 库实现一个 DeepSeek (或其他厂商) 的调用类，注意入参和出参必须和接口保持一致。`”
4. **让系统识别新服务**：
   - 为了让系统认识你写的这个新 Provider，需要检查一下 `manager.py` 中 Provider 的加载机制。
   - 在当前仓库中，更常见的方式是：将新模块放入 `app/providers/` 后，通过重启服务或调用相关 reload 接口，让系统重新扫描并识别你的 Provider，而不是手工往某个固定列表里添加一项。
5. **测试验证**：
   - 系统启动后，确认你的新 Provider 已经出现在 Provider 列表中。
   - 然后创建一个使用该 Provider 的 **Model**。
   - 再创建一个绑定该 `model_id` 的 Agent，发起对话试试看。
   - 请注意：当前系统通常不是“Agent 直接绑定 Provider”，而是“先创建 Model，再让 Agent 绑定这个 Model”。
   - 如果你暂时没有真实厂商账号，也可以使用**兼容 OpenAI 协议的本地假服务**作为有效 Provider 验证方式；这同样算作本实验中可接受的成功扩展路径。另外，若你希望测试真实在线服务，也可以自行了解**魔搭社区**等平台提供的免费 API 调用额度。
6. **🤔 反思题**：
   如果你在新增这个 Provider 的时候，发现自己**不得不去修改 `app/api/v1/endpoints/agents.py` 里面关于模型判断的 `if-else` 分支**才能让它跑通，那么说明原来的代码违反了面向对象的什么设计原则？（提示：O__ Principle）

**本任务建议优先达成**：让系统成功识别你的新 Provider，创建一个使用该 Provider 的 Model，并完成一次模型测试或 Agent 对话验证。

### 任务 2：防具设计（ADT 的实战）
在 `agents.py` 或其他长流程中，我们经常能看到类似 `messages = [{"role": "user", "content": "..."}]` 这样的普通列表被传来传去。这种裸露的数据结构极易被沿途的代码不小心修改。我们将设计一个强类型的 ADT 来穿上“防具”。

**详细操作步骤**：
1. **定义新的上下文环境**：
   - 在项目中合适的位置（例如新建 `app/domain/`，或放在其他职责清晰的位置）创建一个名为 `session_context.py` 的文件。
   - 定义一个类 `class SessionContext:`。
   - 建议你优先围绕当前主聊天链路中的 `chat_request.messages` 和 `final_messages` 来设计这个 ADT，而不是脱离实际代码单独造一个类。
2. **声明 ADT 的灵魂（注释非常重要！）**：
   在类定义的下方，用多行字符串（Docstring）明确写出：
   - **Abstract Function (AF)**：说明这个类用来代表什么（例如：*“代表一次智能体对话的完整短期记忆上下文，包含了历史消息列表及部分系统设定”*）。
   - **Representation Invariant (RI)**：说明内部数据的合法规则（例如：*“`_messages` 列表的长度不得超过 50 条；每条消息的 role 只能是 'user', 'assistant', 或 'system' 其中之一”*）。
3. **实现构造函数与状态保护**：
   - 编写 `__init__` 函数接收初始数据。
   - 为了保护数据，使用私有变量（如 `self._messages`）。
   - 不要直接保存外部传入列表的引用。
4. **防御性编程（Defensive Programming）**：
   - **设值时防守**：编写一个 `add_message(self, message: dict)` 方法。在将 `message` 追加入内部列表前，使用 `copy.deepcopy()` 将其复制一份，或者验证其内容合法性。
   - **取值时防守**：编写一个 `get_messages(self)` 取值方法。千万不要直接 `return self._messages`！（如果是返回引用，外部拿到后依然可以 append 修改）。必须返回列表的**切片/复制**（如 `return copy.deepcopy(self._messages)`）或只读视图。
5. **编写自查方法 `check_rep()`**：
   - 写一个内部方法 `def _check_rep(self):`，在这个方法里使用 `assert` 语句或者抛出异常来校验刚才你在 RI 里写的那些规则。
   - 在 `__init__` 和每一次改变内部状态的方法的最后，都默默调用一次 `self._check_rep()`。
6. **把它接入真实流程**：
   - 不要只定义一个孤立的 `SessionContext` 类然后放在仓库里。
   - 请至少替换 `agents.py` 中**一段 `final_messages` 的构造逻辑**，例如用它接管系统提示词、文件上下文或历史消息中的一部分装配过程。这样可以避免把 `SessionContext` 做成一个孤立类交差，也能把改造范围控制在一个可完成的最小切口内。
7. **单元测试大破坏**：
   - 在 `tests/` 下写一个针对它的测试类；如果仓库中还没有 `tests/` 目录，请先新建。
   - 故意向 `add_message` 传入错误的角色格式，或者故意获取 `get_messages()` 后尝试对其 `append(非法数据)`，测试它是否能拦截并抛出错误（保证自己的内部数据不被污染）。

**本任务建议优先达成**：定义 `SessionContext`（或同等作用的 ADT），至少替换 `agents.py` 中一段 `final_messages` 的构造逻辑，并通过测试证明内部状态得到了有效保护。

## 4. 实验报告及进阶挑战（评分标准）

在对照评分标准之前，建议你先自查三件事：新的 Provider 是否已经被系统识别；`SessionContext` 是否已经进入一段真实消息装配流程；你的测试或运行结果是否已经能够证明改造确实生效。

本实验通过对您重写和改造代码的架构优美程度和稳定性进行阶梯化评分：

*   **及格基础（60分）**：
    *   理解并成功实现了一个新的 Model Provider，项目能成功按照新 Provider 发起模型请求，体现在报告截图和提交代码中。
    *   能证明系统已经识别该 Provider，并完成了“Provider -> Model -> Agent -> 对话”的基本验证链路。
    *   定义了 `SessionContext` 类的基本结构。
*   **工程化标准（80分）**：
    *   达到60分标准。
    *   深度应用 ADT 规范，`SessionContext`（或您选择的其他核心数据结构）拥有完备的 RI、AF 声明以及防御性编程实现机制。
    *   在 `tests/` 下编写了“破坏性测试”（故意传入非法状态、尝试越权修改被暴露的列表），证明被测类的防御屏障能够抛出合理的异常且内部状态不被破坏。
    *   `SessionContext` 不只是一个孤立的类，而是被接入了一段真实消息装配流程。
*   **架构师潜力（100分）**：
    *   达到80分标准。
    *   以下四个进阶提示**每完成一项加10分**，**任意完成其中2项即可达到满分100分**；如果完成超过2项，可在报告中作为额外亮点展示，但总分上限仍为100分。
    *   **重塑核心抽象**：您不仅仅满足于按照现有的 Provider 格式填空，您在实现时发现了 `ModelProvider` 接口本身存在的设计不合理或抽象泄露情况，您大胆**重构了基类的接口契约**。
    *   **子系统改造**：并且，您不仅仅将 `SessionContext` 做成了一个孤立的类，而是设计了一套通用的“智能体记忆抽象层”（例如分为 `MemoryManager`，提取为可以插拔的长期记忆与短期记忆接口），将 `agents.py` 底部乱七八糟的状态管理逻辑彻底铲除，重构出一套高复用性的独立组件（Component）。
    *   **知识不变量防护（ADT 极致应用）**：设计并实现完全不可变（Immutable）的 `KnowledgeChunk` 对象，替换掉诸如 `app/utils/file_processor.py` 等处在流水线中漫天乱飞的裸露字典（Dict）。通过纯熟的防御性编程，确保文档切块对象从诞生开始的文本纯洁性，杜绝任何中间环节污染或篡改特征数据。
    *   **多态存储解耦（OOP 扩展拔高）**：拔高 RAG（检索增强生成）流程的抽象，运用开闭原则（OCP）设计统一的 `VectorStoreProvider` 接口层，从而将上层应用与底层各类繁杂的向量数据库引擎解耦。并以此要求通过多态实现至少两种子类（例如通过纯 Python List 模拟极简的 `LocalMemoryStore`，与真实落地的 `ChromaProvider` 等），彻底将“依赖倒置”原则付诸实践。

## 5. 温馨提示

在完成实验时，请尽量先确认你所阅读和修改的是当前真正生效的代码路径，而不是仓库中可能仍然残留的旧实现。真实工程里，先找对主路径，再动手改造，往往比“直接开始写代码”更重要。

本实验的重点，是帮助你在真实代码中理解 OOP 与 ADT 的核心思想，而不是把你困在环境配置、旧代码辨认或大规模重构中。如果你在实验过程中遇到仓库中存在新旧实现并存、某些文件看起来像残留代码、或者第三方模型服务暂时无法调用，这些都属于真实工程中常见的情况，不必因此慌张。

如果你暂时没有真实厂商 API，也可以使用**兼容 OpenAI 协议的本地假服务**完成 Provider 验证，评分时，我们更看重你是否理解并落实了抽象、封装与防御性编程的思想，而不是是否接入了某一个特定厂商。

另外，60 分标准视为本实验的核心目标，80 分是工程化进阶，100 分是挑战项。先完成一条清晰、可运行、可解释的最小改造路径，再考虑进一步拔高，通常会比一开始就试图“大改全局”更稳妥。
